# PRD mode detection (Step 0)

Load at Step 0 of every session. This reference runs the mode detection protocol and names the dominant failure cause when the PRD is in Mode C or Mode F.

## 1. The six modes

| Mode | Shape | Starting move |
|---|---|---|
| **A** | Greenfield product, no prior PRD, net-new | Step 1 pre-flight, Problem first |
| **B** | Greenfield feature inside an existing product | Step 1 pre-flight with inherited target user |
| **C** | AI-slop PRD already exists; fix before sign-off | Step 1.5 audit |
| **D** | Existing PRD, living; refresh for new cycle | Step 11 changelog entry; update in place |
| **E** | Pivot: new user / problem / success metric / >50% appetite delta | Fork a new PRD; link to prior |
| **F** | Rescue: build stalled because PRD is underspecified | Name the dominant underspecification cause first |

## 2. Detection questions

Ask the user (or infer from input):

1. Does a PRD already exist for this thing? (Yes -> C, D, E, F candidates. No -> A or B.)
2. If one exists: who wrote it, when, and why is this session happening? (AI-generated and unreviewed -> C. Stale but valid -> D. New direction -> E. Engineering stuck -> F.)
3. If no PRD exists: is this a new product or a feature in an existing product? (New -> A. Feature -> B.)

## 3. Block per mode

### Mode A block

Greenfield product. The PRD will cover: the whole thing. Expect to spend the most time on Problem framing (Step 2) and Target User (Step 3) because there is no inherited context. Consider running an Amazon-style PR-FAQ or a Shape Up-style pitch as a parallel artifact for the top-of-PRD narrative; see [prd-anatomy.md](prd-anatomy.md) section 6.

**Research block to produce at Step 0:**

```
Mode: A (Greenfield product)
Why this mode: no prior PRD exists; the user is initiating a new product
Expected tier destination: Tier 2 minimum; Tier 3 if launching externally
Key risks for this mode: invisible PRD (too generic), solution-first (founder has a feature in mind), assumption soup (no user research yet)
Mitigations: apply substitution test hard; ask for one named user the team has actually spoken with; run PR-FAQ if the idea is novel
```

### Mode B block

Greenfield feature in an existing product. The product's target user and success-metric frame are inherited; do not re-litigate. The PRD adds one feature and shows how it fits the existing framing.

**Research block:**

```
Mode: B (Greenfield feature)
Why this mode: existing product PRD or vision doc exists; adding a feature
Expected tier destination: Tier 2 usually sufficient; Tier 3 for regulated domains or major features
Key risks for this mode: feature-first (adding because the roadmap says so, not because users asked), laundry list (PM bundles too many small wants into one PRD)
Mitigations: inherit the product's target user paragraph verbatim; require a problem paragraph that names a specific user friction, not a strategic pillar; cut to one feature
```

### Mode C block

AI-slop PRD exists. Run Step 1.5 audit before proceeding to Step 2 problem framing. The audit output is the input to remediation.

**Research block:**

```
Mode: C (AI-slop PRD rescue)
Why this mode: a PRD exists that was generated by an AI tool (ChatGPT / Claude / ChatPRD / Gemini / Notion AI) with minimal human editing
Dominant failure mode (named after Step 1.5 audit): [one of: invisible / laundry list / solution-first / assumption-soup / hollow / moving-target]
Expected tier destination: rewrite to Tier 1 first; expand only after Tier 1 passes
Key risks for this mode: partial rewrites that leave AI-slop in low-traffic sections (usually Risks and Non-functional); writing on top of AI-slop rather than starting fresh
Mitigations: quote every failing sentence before rewriting; start from a blank Tier 1 Brief, then port good content from the prior doc
```

### Mode D block

Living PRD refresh. The team shipped v1 (or v1.x) and is running the next cycle. Update in place with a dated changelog entry.

**Research block:**

```
Mode: D (Iteration / refresh)
Why this mode: an approved PRD exists; the team is refreshing for the next cycle or responding to post-launch learnings
Expected tier destination: same tier as the existing PRD, or one tier higher if the refresh is expanding scope
Key risks for this mode: silent edits (moving-target failure), stacking new requirements on top of stale no-gos, losing historical rationale
Mitigations: every edit writes to the changelog; the Out-of-Scope section is re-reviewed; the original appetite and success metrics are marked with outcome ("met," "partially met," "missed")
```

### Mode E block

Pivot. The change is large enough that it is a new PRD, not an edit. Fork; cross-link.

**Research block:**

```
Mode: E (Pivot)
Why this mode: new target user OR new problem OR new success metric OR >50% appetite delta; [state which]
Expected tier destination: new Tier 1 Brief first; do not inherit the prior PRD's tier status
Key risks for this mode: treating the pivot as an edit (discarded historical rationale), half-pivots (new target user on top of old success metrics)
Mitigations: archive the prior PRD as "pivoted," link forward; new Brief starts from blank pre-flight
```

### Mode F block

Rescue. Engineering started building and stopped because the PRD is underspecified. Diagnose the dominant cause first; the research section [research pass section 1, Failure modes](RESEARCH-2026-04.md#1-ai-generated-prd-failure-modes) enumerates the six primary dominant causes:

| Dominant cause | Symptom |
|---|---|
| Problem ambiguity | Engineers cannot tell whether a case is in scope |
| Non-estimable requirements | Engineers cannot size the work |
| Missing non-functional thresholds | Architecture cannot be decided; stack cannot be confirmed |
| Silent scope changes | Engineering built X, PM asked for Y, nobody logged the change |
| Acceptance criteria missing | QA cannot write test plans; "done" is ambiguous |
| Integration / trust-boundary undefined | Cannot build without naming who owns what |

**Research block:**

```
Mode: F (Rescue)
Why this mode: build stalled on an approved PRD
Dominant cause (pick one after interviewing eng): [ambiguity / non-estimable / silent NFR / silent changes / missing AC / undefined trust]
Expected tier destination: upgrade the current PRD to the tier where the cause is addressed (Tier 2 for AC and scope; Tier 3 for NFRs and compliance)
Key risks for this mode: bolt-on fixes that don't propagate (tighten one AC, miss three others), treating the rescue as an edit when it is a pivot
Mitigations: interview one engineer explicitly about "what would have made this buildable"; run the tier audit not just the local fix
```

## 4. What the skill does if mode is ambiguous

Ambiguity is usually between A/B or between C/D.

- **A vs. B:** if the project directory contains any prior PRD artifact (PRD.md, a Confluence link in README, a "product overview" doc), default B. Otherwise A.
- **C vs. D:** if the prior PRD was edited by a human within the last 7 days and has more than 2 named signers, default D. Otherwise C.

When the skill cannot tell, ask the user explicitly. Do not silently default; mode sets the cost and the deliverable.

## 5. Sanity checks before leaving Step 0

- The mode has been declared in writing.
- The mode-specific research block has been produced.
- For Mode C: at least one quoted failing sentence is captured.
- For Mode F: the dominant underspecification cause is named.
- The user has not been asked to re-answer pre-flight questions that the prior PRD already answered (Mode D, E inheritance).

If any of these are missing, stop and complete them. The downstream workflow assumes a clean mode declaration.

## 6. The scale of the skill's work per mode

Rough time budget (the user can override):

| Mode | Typical time to Tier 1 | Typical time to Tier 2 |
|---|---|---|
| A | 2-4 hours | 1-2 days |
| B | 30-60 min | 3-5 hours |
| C | 1-2 hours (audit + rewrite) | 1 day |
| D | 30 min (edit + changelog) | 2-4 hours |
| E | Fork, then A or B | New cycle |
| F | 30 min (diagnose) | 1-2 days (rescue) |

These are order-of-magnitude, not commitments. The PRD's appetite discipline applies to the PRD itself: if the PRD has taken 3x the typical time for its mode, the team is probably spending more time on the document than on the discovery; pause and ask why.

## 7. Mode and the "Fishman / Cagan" question

Adam Fishman's *PRDs are the worst way to drive product progress* and Marty Cagan's *Discovery over documentation* both argue that many PRDs should not exist at all. The Mode detection is also a check on whether the PRD is warranted:

- If Mode B with a 2-week appetite and a 2-engineer team, the PRD may be overkill. Consider a Product Brief (Tier 1 only) and no expansion.
- If Mode A for a single-founder side project, Tier 1 Brief plus a handoff block is probably the ceiling; Tier 3 is over-engineered.
- If Mode F with a 10-engineer team on a regulated domain, Tier 3 is the minimum.

The skill does not refuse to write tiers the user asks for, but it surfaces the mismatch. See [iterate-vs-freeze.md](iterate-vs-freeze.md) section 5 for when a Brief is enough.
